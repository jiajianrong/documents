2.2 使用 window 挂载埋点时间戳

降级方案：在body中第一个script标签前添加埋点

这里没有使用domInteractive时间，因为domInteractive需要等待全部的同步js脚本加载执行完毕

这里也没有使用domContentLoadedEventStart时间，因为domContentLoadedEventStart需要等待非async的所有js脚本，包括同步和defer、type=module类型的脚本加载执行完毕

浏览器对首屏渲染做了优化处理，在解析完主体dom文档后，会优先执行渲染一次页面，放弃等待js执行完毕再渲染

3.3 MutationObserver+fetch

记录初始化fetch请求结束时间，通过window.MutationObserver监听dom变化，JSTracker采用的这个方案

缺点：需要将脚本前置到业务js前加载

  1. 通过window.MutationObserver追踪监听dom结构变化，记录每次dom结构变化时间。

  2. 重写window.fetch，在原生fetch基础上增加请求前和请求后的时间监控处理。

    • 请求前handle:

      • 过滤请求，是否记录当前请求。过滤规则为首次请求发出timeout（默认配置为500ms）后不再抓取请求记录时间，黑名单blackList（默认配置为空数组）内url不抓取记录。
      • 当url的hash变化时重置当前状态为初始状态，保证hash变化时能够抓取到单页面路由切换初始化请求时间。
      • 记录每次请求时间 fetch-start，标记首次请求时间 this.fetchStartTime。

    • 请求后handle:

      • 记录每次请求结束时间（成功、失败）fetch-end，前提为过滤选中的请求，规则为请求前过滤请求规则。

      • 每次请求结束检测是否是首屏最后一个请求

        • 先判断请求记录数是否为偶数，只有当次数为偶数时才能保证当前时间之前的请求已经全部请求完毕。
        • 预设paintingTime（默认值100ms）时间为dom树绘制时间，当该时间过后检查时间轨迹中最后记录时间若是fetch相关时间，或fetch相关记录数为奇数，且检查时间间隔在timeout时间范围内，则递归检测。当条件满足或到截止时间timeout时取时间轨迹中最后一次dom变化时间为目标时间，即首屏渲染结束时间 targetInfo.time。
    • fmp= targetInfo.time - navigationStart

4.TTI - 能用吗？

全称 Time To Interactive，翻译为可交互时间，指的是应用在视觉上都已渲染出来了，完全可以响应用户的输入了。是衡量应用加载所需时间并能够快速响应用户交互的指标。与 FMP 相同，很难规范化适用于所有网页的 TTI 指标定义。

6.FSP

FPS，全称 Frames Per Second，翻译为每秒帧率，表示的是每秒钟画面更新次数，当今大多数设备的屏幕刷新率都是60次/秒。

参考标准：

  • 帧率能够达到 50 ～ 60 FPS 的动画将会相当流畅，让人倍感舒适；
  • 帧率在 30 ～ 50 FPS 之间的动画，因各人敏感程度不同，舒适度因人而异；
  • 帧率在 30 FPS 以下的动画，让人感觉到明显的卡顿和不适感；
  • 帧率波动很大的动画，亦会使人感觉到卡顿
x
var lastTime = performance.now();
var frame = 0;
var lastFameTime = performance.now();
var loop = function (time) {
    var now = performance.now();
    var fs = (now - lastFameTime);
    var fps = Math.round(1000 / fs);
    frame++;
    
    // 防止2次渲染帧大于1s的情况统计有误
    if (now - lastTime > 1000) {
        fps = Math.round((frame * 1000) / (now - lastTime));
        lastTime = now;
        frame = 0;
    }
​
    lastFameTime = now;
​
    if (fps < 30) {
        // console.log('//////////// fps //////////', fps);
    }
​

};

Babel的工作流程

@babel/core: babylon && babel-traverse && babel-generator

Babel三个阶段：解析parse | 转换transform | 生成generate

  • 解析

    将代码解析成抽象语法树（AST-abstract syntax tree），Babel是通过Babylon实现的。在解析过程中有两个阶段：词法分析(类似生成 dom 节点) 和 语法分析(类似生成 dom 树) 。词法分析阶段把字符串形式的代码转换为令牌（tokens）流，令牌类似于AST中节点；而语法分析阶段则会把一个令牌流转换成 AST的形式，同时这个阶段会把令牌中的信息转换成AST的表述结构。

  • 转换

    在这个阶段，Babel接受得到AST并通过babel-traverse对其进行深度优先遍历，在此过程中对节点进行添加、更新及移除操作。这部分也是Babel插件介入工作的部分。（一般插件在此搞事情）

  • 生成

    将经过转换的AST通过babel-generator再转换成js代码，过程就是深度优先遍历整个AST，然后构建可以表示转换后代码的字符串。

 

babylon并非由babel团队自己开发的，而是fork的acorn项目，acorn也是webpack的AST解析器

我的题解

利用babel提供的AST解析功能，在转换阶段找到代码中的所有方法，在函数体外包裹一层try-catch

  1. 分析观察AST树：https://astexplorer.net
  2. 找到所有与Function相关节点
  3. 替换节点body，操作节点使用babel-template、babel-types库（类似lodash）
